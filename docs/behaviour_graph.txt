Behavior Graph Specification (BGS)
1. Purpose
The Behavior Graph Specification (BGS) defines the complete behavioral modeling framework for the Self-Observing Testing System (SOTS).
This document establishes:
* Behavioral graph architecture
* State modeling standards
* Transition modeling standards
* Workflow modeling standards
* Graph construction algorithms
* Graph storage architecture
* Graph lifecycle management
* Graph analytics capabilities
* Phase-based graph evolution
* Behavioral intelligence foundations
The Behavior Graph is the central intelligence model of SOTS and serves as the primary representation of how software behaves in practice. It transforms raw events, sessions, and workflows into a continuously evolving behavioral map of an application.
2. Scope
This specification applies to:
Phase 1 — Behavioral QA Platform
Behavior graphs generated from:
* Recorded sessions
* Demonstration workflows
* SDK-captured interactions
Phase 2 — Production Intelligence Platform

The graph model must support all present and future intelligence capabilities without redesign.

Behavior Graph Specification — Extension
Section 22: Declared Intent Graph & Flow Reconciliation
22.1 Purpose
Sections 1–21 of this specification define the Observed Behavior Graph: a graph constructed bottom-up from recorded sessions, where every node and edge exists because it was actually witnessed by the platform. This section introduces a second graph type, the Declared Intent Graph, constructed top-down by a QA Engineer or Product Manager during organization/application onboarding, before any telemetry exists.
The two graphs are reconciled continuously rather than merged. The Declared Intent Graph expresses what the application should do. The Observed Behavior Graph expresses what the application has been seen doing. The gap between them — in both directions — is the new source of quality intelligence defined in this section.
This capability is named the Flow Declaration & Reconciliation System (FDRS).
22.2 Relationship to Existing Graph Model
FDRS does not replace or modify the Observed Behavior Graph defined in Sections 4–14. It introduces a parallel object using the same core primitives (State, Action, Transition, Workflow — Section 4) so that both graphs remain comparable and storable in the same graph database model (Section 14).
Declared Intent Graph          Observed Behavior Graph
(human-authored, pre-traffic)  (system-derived, post-traffic)
        │                              │
        └──────────────┬───────────────┘
                        ▼
              Reconciliation Engine
                        │
                        ▼
              Three-Way Classification
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
  CONFIRMED         TRUE GAP        UNDECLARED
  (declared AND   (declared AND   (observed AND
   observed)       not observed)   not declared)


A node or transition in the Declared Intent Graph carries a provenance field distinguishing how it entered the graph:
{
  "stateId": "state_044",
  "stateName": "PAYMENT_FAILED",
  "graphType": "DECLARED",
  "provenance": "SUGGESTED_ACCEPTED",
  "declaredBy": "user_id",
  "confidence": null
}


Valid provenance values:
* USER_AUTHORED — typed or drawn directly by the QA/PM during declaration.
* SUGGESTED_ACCEPTED — proposed by the Derivation Engine (22.4) and accepted by the user.
* SUGGESTED_REJECTED — proposed and explicitly rejected. Retained (not deleted) so the system does not re-suggest it and so rejection patterns can improve future suggestions.
* DEMONSTRATION_PROMOTED — originated as an Observed-graph node and was later promoted into the Declared graph because a developer confirmed it during demonstration review (see 22.7).
Nodes in the Observed Behavior Graph keep their existing provenance model unchanged (Section 11, Section 12); FDRS does not alter how the Observed graph is built.

22.4 Derivation Engine
22.4.1 Purpose
As each node is added to a Declared Intent Graph, the Derivation Engine proposes additional states, transitions, and branch flows the user likely intends but has not explicitly entered — failure paths, edge states, recovery flows, and category-appropriate variants (Section 5.2 categories: Navigation, UI, Business, Error, System).
22.4.2 Two-Tier Lookup Strategy
Node Added
    │
    ▼
Tier 1: Internal Pattern Library
    │
    ├── Match found, sufficient confidence ──→ Suggest
    │
    └── No match / thin coverage
            │
            ▼
    Tier 2: External Search Enrichment
            │
            ▼
    Candidate filtered, normalized, suggested


Tier 1 — Internal Pattern Library
A maintained, versioned library of QA patterns keyed by state category and semantic name, independent of any tenant's data. This is curated content (analogous to the "generic QA packs" referenced in Phase 1.5B: auth, crud, api, errors, permissions, search, upload, empty-states, recovery, loading) rather than a live model. Example entry:
{
  "patternId": "pat_login_001",
  "triggerStateNames": ["LOGIN", "SIGN_IN", "AUTHENTICATE"],
  "triggerCategory": "BUSINESS",
  "suggestedBranches": [
    {"stateName": "LOGIN_FAILURE", "category": "ERROR"},
    {"stateName": "ACCOUNT_LOCKED", "category": "ERROR"},
    {"stateName": "PASSWORD_RESET_REQUESTED", "category": "BUSINESS"},
    {"stateName": "SESSION_TIMEOUT", "category": "SYSTEM"}
  ],
  "source": "INTERNAL_LIBRARY",
  "libraryVersion": "1.4.0"
}


Tier 1.5 — Cross-Tenant Structural Pattern Match
In addition to the static library, the Derivation Engine may query a structural pattern index built from anonymized graph shapes across all tenants' Declared and Observed graphs. This index stores only state names, categories, and transition topology — never tenant identifiers, application names, or any data that could re-identify a source organization.
FDRS-PRIV-001 Cross-tenant structural matching shall never reveal, directly or indirectly, which tenant a pattern originated from.
FDRS-PRIV-002 Cross-tenant structural matching shall operate only on state/category/transition shape data. Metadata, custom field names, business-specific terminology beyond standard category labels, and any payload content are excluded from the structural index.
FDRS-PRIV-003 A tenant's Declared or Observed graph data shall contribute to the structural index only in aggregate, frequency-weighted form (e.g. "73% of CHECKOUT flows include an OUT_OF_STOCK branch"), never as a retrievable individual record.
FDRS-PRIV-004 Participation in the cross-tenant structural index is part of standard platform operation, consistent with how the platform already aggregates non-identifying behavioral patterns; tenants requiring full exclusion may request it through organization-level configuration.
Tier 2 — External Search Enrichment
Used only when Tier 1 (library + structural index) returns no match or low-confidence results for a novel state name or category combination. External search results are used to enrich the suggestion candidate pool, not surfaced verbatim — results are normalized into the same suggestedBranches structure as Tier 1 before being shown to the user, and any matched content is paraphrased rather than quoted, consistent with platform-wide content handling standards.
FDRS-DERIVE-001 External search shall be invoked only after internal lookup (Tier 1 / 1.5) is exhausted for a given node. FDRS-DERIVE-002 Suggestions surfaced to the user shall always be labeled with their source tier (Library, Cross-Tenant Pattern, or External Enrichment) so the user can weight their trust accordingly. FDRS-DERIVE-003 The Derivation Engine shall never auto-add a suggested node to the Declared Intent Graph without explicit user acceptance.
22.4.3 Suggestion Output Model
{
  "suggestionId": "sug_881",
  "parentStateId": "state_012",
  "suggestedStateName": "PAYMENT_FAILED",
  "category": "ERROR",

22.5 Reconciliation Engine
22.5.1 Purpose
Once telemetry begins flowing (via demonstration sessions or, in later phases, production traffic), the Reconciliation Engine continuously diffs the Declared Intent Graph against the Observed Behavior Graph for the same application.
Supports a generalization of Coverage Analysis (Section 9 of the Data Flow Document; FR-035 through FR-041) and Missing State Detection (Section 10 of the Data Flow Document; FR-042 through FR-046), but operates against a human-declared baseline rather than the graph's own internal completeness.
22.5.2 Three-Way Classification
               Observed?
                Yes        No
Declared? Yes  CONFIRMED   TRUE_GAP
          No   UNDECLARED  (not applicable)


CONFIRMED — State/transition exists in both graphs. This is genuine, evidence-backed coverage.
TRUE_GAP — Declared but never observed. Stronger signal than the existing Missing State/Flow detection, because a human explicitly asserted this should be reachable; it is not an inference from absence.
UNDECLARED — Observed but not in the Declared Intent Graph. The application is doing something nobody specified. This may indicate: an incomplete declaration, an unintended code path, a legitimate flow the declarer forgot, or risk-worthy emergent behavior.
FDRS-RECON-001 The Reconciliation Engine shall classify every node and transition in either graph into exactly one of CONFIRMED, TRUE_GAP, or UNDECLARED. FDRS-RECON-002 UNDECLARED classifications shall not be treated as errors by default; they shall be surfaced for human review as a distinct report category (22.5.3), not folded into existing Missing State/Flow reports. FDRS-RECON-003 Reconciliation shall re-run incrementally as new sessions are processed, not only once at flow-completion time.
22.5.3 Reconciliation Report
A new report type, extending the QA Report Specification's report classification model (QRS Section 4) as a sibling to Coverage Reports and Missing Flow Reports.
FLOW RECONCILIATION REPORT

FDRS-RPT-001 The Reconciliation Report shall identify, for every True Gap, whether the originating node was USER_AUTHORED or SUGGESTED_ACCEPTED, to distinguish "the QA team specifically asked for this and it's missing" from "the system suggested it and it turned out to not exist yet." FDRS-RPT-002 The Reconciliation Report shall support the same export formats as other QA reports (QRS Section 24: PDF, CSV, JSON, HTML).
22.6 Ruleset Compilation
22.6.1 Purpose
Once a declared workflow is marked complete (22.3), it compiles into a per-application Ruleset — the artifact that downstream review (demonstration walkthroughs, future production review) is checked against.
Generic QA Best-Practice Rules
   (naming conventions, comment standards,
    universal edge-case patterns)
              +
Declared-Flow-Derived Rules
   (this application's specific expected
    states, transitions, workflows)
              +
Accepted-Suggestion Rules
   (branches the Derivation Engine proposed
    and the user kept)
              =
        Application Ruleset


{
  "rulesetId": "rs_app_004",
  "applicationId": "app_004",
  "version": "1.0",
  "compiledAt": "timestamp",
  "sourceFlows": ["wf_checkout", "wf_registration"],
  "ruleCount": 47
}


FDRS-RULE-001 A Ruleset shall always include the platform's generic QA best-practice rules regardless of whether any flow has been declared. FDRS-RULE-002 A Ruleset shall be recompiled whenever its source Declared flows change (new flow completed, existing flow edited and resaved). FDRS-RULE-003 Rulesets shall be versioned; prior versions shall remain available for historical reconciliation against sessions recorded under that version.
22.6.2 Ruleset Application
The compiled Ruleset is consumed by:
* The Reconciliation Engine (22.5), to know what "declared" means per workflow.
* The existing Missing State Engine and Coverage Engine, which may use generic-rule entries to flag gaps even for undeclared flows.
* The demonstration review surface (22.7), to point out gaps live as a developer or QA engineer walks through the application.
22.7 Interaction with Developer Demonstration Mode
Sections 22.3–22.6 describe declaration happening before demonstration. In practice, the two are designed to inform each other continuously, not strictly sequentially.
Declared Intent Graph  ──informs──▶  Demonstration Review
        ▲                                    │
        │                                    │
        └──────── promotion (22.2) ──────────┘
                  (developer confirms an
                   observed-but-undeclared
                   state belongs in the
                   intended flow)


During a demonstration session, if the developer's walkthrough produces an UNDECLARED node (22.5.2), the dashboard may prompt the developer: "This state wasn't in your declared flow — is it expected?" An affirmative response promotes the node into the Declared Intent Graph with provenance: DEMONSTRATION_PROMOTED (22.2) and triggers Ruleset recompilation (22.6.1).
FDRS-DEMO-001 Demonstration-time promotion of an Observed node into the Declared Intent Graph shall require explicit developer confirmation; it shall never happen automatically. FDRS-DEMO-002 The existing Behavioral Graph Generation pipeline (BGS Sections 10–12) is unaffected by FDRS; demonstration sessions continue to build the Observed Behavior Graph exactly as specified, independent of whether a Declared Intent Graph exists.
22.8 Manual Editing & Correction
FDRS-EDIT-001 A Declared Intent Graph shall remain editable by users with appropriate role permissions (Developer, QA Engineer, Product Manager per SEAD Section 7.3) after initial completion. FDRS-EDIT-002 Manual edits to a completed flow shall trigger Ruleset recompilation (22.6.1) and shall be logged with the same audit standard as other security-sensitive configuration changes (SEAD Section 12). FDRS-EDIT-003 The system shall never overwrite a USER_AUTHORED node based on Observed-graph evidence; conflicting evidence is surfaced via the Reconciliation Report (22.5.3) as UNDECLARED, leaving the decision to the human.
22.9 Governance Rules
FDRS-GOV-001 Every Declared Intent Graph node must carry a provenance value (22.2). FDRS-GOV-002 Declared and Observed graphs must remain independently queryable; reconciliation is a derived view, not a merged storage model. FDRS-GOV-003 Rejected suggestions (SUGGESTED_REJECTED) must be retained, not deleted, to prevent re-suggestion loops and to support future improvement of the Derivation Engine. FDRS-GOV-004 Cross-tenant structural pattern data must satisfy FDRS-PRIV-001 through FDRS-PRIV-004 at all times, including during platform scaling and future Phase 2/3 enhancements. FDRS-GOV-005 FDRS must remain optional at the application level; an application with no Declared Intent Graph must continue to function exactly as specified in BGS Sections 1–21, with Coverage and Missing State analysis operating purely against the Observed Behavior Graph as today.
22.10 Scope Note — Relationship to MVP Scope Document
This capability is intended for Phase 1 / MVP inclusion per product direction. This creates a direct tension with the existing MVP Scope Document (MSD Section 9), which explicitly excludes "AI workflow generation" and "AI recommendations" from Phase 1, citing insufficient data maturity. The Derivation Engine's suggestion behavior — proposing branch states a human did not type — functions similarly to the excluded category even though it is implemented as a curated pattern library plus structural matching rather than a learned model.
This section does not resolve that tension; it is flagged here for explicit product/business sign-off before implementation, since it changes what MSD Section 1 calls "the authoritative source for Phase 1 feature inclusion and exclusion decisions." If approved, MSD Section 7 should be amended with a new Feature Group for Flow Declaration & Reconciliation, and MSD Section 9's AI Features exclusion should be narrowed to explicitly carve out "curated/structural suggestion, as distinct from learned/generative AI recommendation."
